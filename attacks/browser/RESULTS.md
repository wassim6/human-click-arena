# Results — selenium / seleniumbase / puppeteer / playwright vs the detector

Generated by `run_all.py`. Raw data: [`results.json`](results.json); the exact
traces that were scored are under [`captured/`](captured/). 32 runs total: 4
engines × 3 strategies × {plain, stealth} at dpr=1, plus a dpr=2 humanized
cross-section.

## How it was run

The scorer (`server/scorer.py`) and the real `client/collector.js` are served by
`harness_server.py` on a fixed-target page. Each engine moves to the target and
clicks; the collector captures the pointer trace exactly as the live demo would,
and it is POSTed to `/score`.

- **Playwright** and **Puppeteer** ran **live** against a real headless Chromium
  (Puppeteer's `humanized` uses the real **ghost-cursor** library).
- **Selenium** and **SeleniumBase** were measured on **linux-arm64**, where no
  official `chromedriver` exists, so the real driver can't launch
  (`NoSuchDriverException`). Their **exact** pointer event stream — integer-only
  W3C Actions coordinates — was replayed through the same Chromium and scored
  (`emulate.py`). On x86_64/macOS the shipped `selenium_attack.py` /
  `seleniumbase_attack.py` produce the same trajectories with the genuine
  libraries. The trajectory is the only thing the scorer sees, and every
  WebDriver/CDP click reduces to the same pointer primitive.

Verdict bands: `score >= 0.5` → **human**, `0.3–0.5` → **suspicious**, `< 0.3` → **bot**.

## What it shows

**1. Stealth makes no behavioral difference — at all.** Every `--stealth` row
matches its plain twin. The plugins do their job at the fingerprint layer
(`navigator.webdriver` flips `true → false` in all four engines), but the score
is byte-for-byte the same band because none of them change the *shape* of the
motion. This is the central claim of the project, confirmed across all four
stacks: the detector never looked at the framework, so hiding the framework buys
nothing.

**2. The movement strategy is the entire game.**
- `native` `.click()` is an instant red flag for every engine: it emits a single
  hop to the element centre, so there is no movement to judge and the hard gate
  fires (score `0.00`, "no pointer movement before click"). Most scrapers click
  exactly this way.
- `linear` (a straight, evenly-timed glide) is caught by the straightness +
  easing-fit signals, scoring `0.15–0.42`. It lands right on the scorer's
  "matches a known tween" cap, which is why it wobbles between **bot** and
  **suspicious** (see caveat below).
- `humanized` (curved Bezier + jitter + overshoot/correct; ghost-cursor for
  Puppeteer) **defeats the behavioral layer**, scoring **human 0.56–0.78** on a
  normal display. This is exactly the "honest ceiling" the top-level README
  predicts: a determined attacker who humanizes the trajectory beats the
  client-side behavioral scorer alone.

**3. The HiDPI sub-pixel tell is the one structural catch.** On a `dpr = 2`
display, humanized Playwright and Puppeteer still pass (they move to *fractional*
pixels, `int_coord_ratio ≈ 0.01–0.03`), but humanized **Selenium and
SeleniumBase are capped to bot `0.22`** — the W3C Actions API only accepts
**integer** coordinates, so every sample lands on a whole pixel
(`int_coord_ratio = 1.0`) and the HiDPI gate fires regardless of how human the
path is. This is the most valuable defensive signal in the whole scorer: the
single most common automation API (WebDriver Actions, used by Selenium and
SeleniumBase) *structurally cannot* produce sub-pixel motion, and on HiDPI that's
unfakeable in JS.

## Recommendations to harden the scorer

- **Tighten the linear/easing cap.** The cap `easing_r2 >= 0.985 and directness
  >= 0.99` is brittle: identical linear glides scored `0.15` and `0.42` across
  runs because timing noise nudged `easing_r2` across `0.985`. Lower the
  threshold (e.g. `0.97`) or fold easing-fit in as a smooth penalty instead of a
  hard cliff so a straight line can't ever reach "suspicious".
- **Lean on the HiDPI integer tell, but don't over-trust it.** It cleanly stops
  WebDriver-Actions engines on HiDPI, yet real users on 1× displays also produce
  integers, so keep it gated on `dpr > 1` (as it already is) and pair it with the
  shape signals.
- **Add shape signals that Bezier humanizers don't reproduce.** The humanized
  paths win mainly on curvature + jitter + a single overshoot. Consider:
  monotonic-progress / "too efficient" detection, velocity-autocorrelation,
  stop-and-go dwell distribution, and a pre-click micro-pause check — ghost-cursor
  and hand-rolled Bezier are smoother and more monotonic than a real hand.
- **Feed these back as training data.** The eight humanized traces in
  `captured/` are labeled-bot examples that currently score **human** — precisely
  the regression fixtures the arena is meant to collect. The natural next step is
  to drop them into `tests/fixtures/` and `bypasses/`.

## Reproduce

```bash
cd attacks/browser
pip install -r requirements.txt && python -m playwright install chromium && npm install
python run_all.py                                   # the dpr=1 matrix
python run_all.py --engines selenium,playwright --strategies humanized --dpr 2
```
